Program Development Life Cycle

The structured, iterative process used by programmers to turn a real-world problem into a working, tested program — examined heavily on Paper 2.

The Four Stages at a Glance

Stage 1
Analysis

Understand the problem. Gather requirements. Identify inputs, processes, outputs and storage.

Stage 2
Design

Plan the solution using structure diagrams, flowcharts and pseudocode.

Stage 3
Coding

Translate the design into a high-level programming language (Python, Java, VB).

Stage 4
Testing

Run the program with carefully chosen test data to find and fix errors.

Stage 1 — Analysis (in depth)

Analysis

Analysis is about understanding the problem fully before writing any code. It involves four key activities:

  • Requirements gathering — interview the client/user, study the existing system, read documents, identify what the program must do (functional requirements) and how well it must do it (non-functional, e.g. speed, security).
  • Abstraction — keeping only the relevant details and ignoring/hiding the rest. E.g. for a payroll program you keep pay rate and hours worked; you ignore the employee's eye colour.
  • Decomposition — breaking the big problem into smaller, manageable sub-problems that can each be solved separately.
  • Identifying Inputs / Processes / Outputs / Storage — every program is essentially I → P → O, often with files/databases for persistent storage.

Stage 2 — Design (in depth)

Design

Design takes the analysis output and produces a plan that a programmer can follow. Three standard tools are examined:

Structure diagram

A top-down, hierarchical tree showing how the whole system is broken into modules and sub-modules.

Use when: planning overall architecture at the start of design.

Flowchart

A visual diagram of the algorithm using standard symbols (start/stop, process, decision, input/output).

Use when: showing flow of control, especially loops and decisions.

Pseudocode

Text-based, language-independent description of the algorithm using keywords like IF, WHILE, FOR.

Use when: writing detailed step-by-step logic just before coding.

Sample pseudocode (login validation)

INPUT username
INPUT password
IF username = "admin" AND password = "1234" THEN
   OUTPUT "Access granted"
ELSE
   OUTPUT "Access denied"
ENDIF

Stage 3 — Coding (in depth)

Coding

Coding is the translation of the design into a high-level programming language such as Python, Java or Visual Basic. A high-level language is chosen because it is easier to read, write and maintain than machine code.

  • Modular coding — splitting the program into procedures (which carry out a task) and functions (which carry out a task and return a value). Each module corresponds to one box on the structure diagram.
  • Benefits of modules: easier to test, easier to debug, can be reused, can be written by different team members in parallel, easier to maintain.
  • Code conventions: meaningful identifiers, comments, indentation, constants for fixed values.

Example — modular pseudocode

PROCEDURE Greet(name : STRING)
   OUTPUT "Hello ", name
ENDPROCEDURE

FUNCTION Square(x : INTEGER) RETURNS INTEGER
   RETURN x * x
ENDFUNCTION

// Main program
Greet("Alex")
OUTPUT Square(7)

Stage 4 — Testing (in depth)

Testing

Testing checks that the program does what it should and copes with bad input. Four kinds of test data are defined by the syllabus.

Normal

Sensible values well inside the accepted range. Should be accepted.

Range 1–100 → 42

Boundary

Values on the edge of the accepted range. Largest accepted & smallest accepted.

Range 1–100 → 1 or 100

Extreme

The most extreme valid values at each end of the accepted range (often same as boundary).

Range 1–100 → 1 and 100

Erroneous

Values the program should reject — wrong type or out of range.

Range 1–100 → -5, 101, "abc"

Boundary vs Extreme — key distinction: Boundary data tests just inside and just outside the limit (e.g. 0, 1, 100, 101). Extreme data tests the largest and smallest values that are still accepted (1 and 100). In the IGCSE answers, the boundary just inside and the extreme can be the same number.

The Cycle is Iterative

The four stages form a cycle: bugs send you back to coding, design flaws send you back to design, and missed requirements send you all the way back to analysis.

Types of Programming Errors

Three classes of error appear on Paper 2 — make sure you can define each and give an example.

Syntax error

A break of the rules (grammar) of the programming language. Detected by the compiler or interpreter before/while the program runs — the program will not execute.

Example:

IF x > 10      // missing THEN
   OUTPUT "big"
ENDIF

Or in Python: print("hi" — missing closing bracket.

Logic error

The program runs without crashing but produces the wrong output. The instructions are valid but they do the wrong thing.

Example: an average function that uses total / 2 instead of total / n, or IF age > 18 instead of IF age >= 18.

Runtime error

The program starts running but crashes (aborts) during execution because it tries to do something impossible.

Example: division by zero (x / 0), accessing array element a[20] when array only has 10 elements, or trying to open a file that does not exist.

Computational Thinking

The three problem-solving skills every IGCSE candidate must be able to define, apply and exemplify: abstraction, decomposition and algorithmic thinking.

The Three Skills

Abstraction

Removing or hiding unnecessary detail so that only the information needed to solve the problem remains.

Example: the London Tube map shows stations and lines but ignores exact distances, curves and surface geography — those details are irrelevant for finding a route.

Decomposition

Breaking a large complex problem into smaller, more manageable sub-problems that can be solved one at a time.

Example: a quiz app decomposes into: user login, question bank, scoring engine, results screen and user interface.

Algorithmic thinking

Working out a clear, ordered sequence of steps (an algorithm) that solves the problem and can be repeated reliably.

Example: writing the precise steps to make a cup of tea — kettle on, wait for boil, pour into cup with teabag, wait, remove bag, add milk.

Worked Example — Vending Machine Controller

Suppose we are asked to build the software that controls a vending machine. We apply decomposition to break the problem down, then abstraction to keep only the relevant details, then algorithmic thinking for each module.

Top-level decomposition (structure diagram)

flowchart TD A[Vending Machine] --> B[Accept Money] A --> C[Select Product] A --> D[Dispense Product] A --> E[Give Change] B --> B1[Read Coin] B --> B2[Validate Coin] B --> B3[Add to Balance] C --> C1[Display Menu] C --> C2[Read Selection] C --> C3[Check Stock] D --> D1[Activate Motor] D --> D2[Update Stock] E --> E1[Calculate Change] E --> E2[Release Coins]

Abstraction in this problem

We ignore: the colour of the machine, the brand of the cans, the music it plays, the weight of the coins. We keep: coin value, product ID, product price, stock level, balance held, change owed.

Algorithm for "Accept Money" (pseudocode)

balance ← 0
WHILE balance < price DO
   INPUT coin
   IF ValidCoin(coin) = TRUE THEN
      balance ← balance + coin
   ELSE
      OUTPUT "Coin rejected"
   ENDIF
ENDWHILE
OUTPUT "Balance reached"

Each leaf box in the structure diagram becomes a small procedure or function during the coding stage — a direct link between Computational Thinking and the Program Development Life Cycle.

Structure Diagrams Explained

A structure diagram is the most common way to present decomposition on paper. Key features:

  • Top-down — the whole system sits at the top; the diagram becomes more detailed as you go down.
  • Hierarchical — boxes are organised in levels (parent → child), with each child being a sub-task of its parent.
  • Modules, not code — each box describes what is done, not how. Algorithms come later (in pseudocode/flowcharts).
  • Planning tool — used during the Design stage of the life cycle to give the team an overview before writing detailed algorithms.
  • Reusable modules — a module that appears in more than one place (e.g. "Validate Coin") can be written once and reused.

Difference from a flowchart: a structure diagram shows what modules exist and how they relate; a flowchart shows the order in which steps execute, including loops and decisions.

1. Variables & Constants

Declare every variable before use with DECLARE. Use ← for assignment, never =. Constants are fixed at declaration.

// Declaration: DECLARE name : TYPE
DECLARE Counter : INTEGER
DECLARE Price : REAL
DECLARE Initial : CHAR
DECLARE Name : STRING
DECLARE Found : BOOLEAN
DECLARE Birthday : DATE

// Assignment uses the left arrow
Counter ← 0
Price ← 4.99
Initial ← 'A'
Name ← "Alex"
Found ← FALSE
Birthday ← 19/05/2026

// Constants — value fixed at declaration
CONSTANT Pi = 3.14159
CONSTANT MaxSize = 100
CONSTANT Greeting = "Hello"

Tip Identifiers are case-sensitive in markschemes — stay consistent.

2. Input / Output

INPUT reads from the user; OUTPUT writes to the screen. Strings are joined with commas in OUTPUT lists.

// Plain input
INPUT Name
INPUT Age

// Input with a prompt (Cambridge accepts both forms)
OUTPUT "Enter your name: "
INPUT Name

// Simple output
OUTPUT Name
OUTPUT "The answer is ", Answer

// Multi-item output (comma-separated list)
OUTPUT "Hello, ", Name, "!"
OUTPUT "Score: ", Score, " out of ", Total

// Worked snippet
OUTPUT "Enter age: "
INPUT Age
OUTPUT "You are ", Age, " years old."

Note OUTPUT automatically moves to a new line at the end.

3. Selection (IF & CASE)

Two structures: IF…THEN…ELSE…ENDIF for binary/nested choices, CASE OF for many discrete values.

// Simple IF
IF Age >= 18 THEN
   OUTPUT "Adult"
ENDIF

// IF…ELSE
IF Mark >= 50 THEN
   OUTPUT "Pass"
ELSE
   OUTPUT "Fail"
ENDIF

// Nested IF — grade boundaries
IF Mark >= 80 THEN
   Grade ← 'A'
ELSE
   IF Mark >= 70 THEN
      Grade ← 'B'
   ELSE
      IF Mark >= 60 THEN
         Grade ← 'C'
      ELSE
         IF Mark >= 50 THEN
            Grade ← 'D'
         ELSE
            Grade ← 'F'
         ENDIF
      ENDIF
   ENDIF
ENDIF

// CASE — cleaner alternative
CASE OF Grade
   'A' : OUTPUT "Excellent"
   'B' : OUTPUT "Very good"
   'C' : OUTPUT "Good"
   'D' : OUTPUT "Pass"
   'F' : OUTPUT "Fail"
   OTHERWISE : OUTPUT "Invalid grade"
ENDCASE

// CASE with ranges and lists
CASE OF Mark
   80 TO 100 : Grade ← 'A'
   70 TO 79  : Grade ← 'B'
   60 TO 69  : Grade ← 'C'
   50 TO 59  : Grade ← 'D'
   OTHERWISE: Grade ← 'F'
ENDCASE

4. Iteration — three loop types

Cambridge expects you to choose the correct loop. FOR for known counts, WHILE for pre-test, REPEAT for post-test.

// FOR — count-controlled (known number of iterations)
FOR i ← 1 TO 10
   OUTPUT i
NEXT i

// FOR with STEP — count downwards or in fixed jumps
FOR i ← 10 TO 1 STEP -1
   OUTPUT i
NEXT i

FOR i ← 2 TO 20 STEP 2
   OUTPUT i        // 2, 4, 6, … 20
NEXT i

// WHILE — pre-test (may run zero times)
Total ← 0
INPUT Number
WHILE Number <> -1 DO
   Total ← Total + Number
   INPUT Number
ENDWHILE
OUTPUT "Total: ", Total

// REPEAT — post-test (always runs at least once)
REPEAT
   OUTPUT "Enter password: "
   INPUT Password
UNTIL Password = "secret123"

4b. When to use which loop

Pick by reading the question wording. Examiners look for the correct construct, not just a working answer.

  • FOR "do this 10 times", "for each pupil in class of 30", "every element of the array" — number of iterations is known.
  • WHILE "keep going while temperature is above 20", "until the user enters STOP" (test first). Condition may already be false ⇒ body skipped.
  • REPEAT "ask until input is valid", validation loops, menu prompts. Body must execute at least once.
  • Switch FOR → WHILE if the count depends on user input that you don't yet have.
// Validation pattern — REPEAT is ideal
REPEAT
   OUTPUT "Enter age (0-120): "
   INPUT Age
UNTIL Age >= 0 AND Age <= 120

// Sentinel pattern — WHILE is ideal
INPUT Value
WHILE Value <> 0 DO
   Sum ← Sum + Value
   INPUT Value
ENDWHILE

5. Arrays (1D & 2D)

Arrays are 1-indexed by default in Cambridge pseudocode. ARRAY[1:n] means indices 1 through n inclusive.

// 1-D array — 30 INTEGER marks
DECLARE Marks : ARRAY[1:30] OF INTEGER

// Assigning into elements
Marks[1] ← 75
Marks[2] ← 82
Marks[30] ← 60

// Reading every element with FOR
FOR i ← 1 TO 30
   OUTPUT "Enter mark ", i, ": "
   INPUT Marks[i]
NEXT i

// Summing array contents
Total ← 0
FOR i ← 1 TO 30
   Total ← Total + Marks[i]
NEXT i
Average ← Total / 30

// 2-D array — 5x5 grid of CHAR
DECLARE Grid : ARRAY[1:5, 1:5] OF CHAR

Grid[1, 1] ← 'X'
Grid[3, 4] ← 'O'

// Nested loop to fill a 2-D array
FOR Row ← 1 TO 5
   FOR Col ← 1 TO 5
      Grid[Row, Col] ← '.'
   NEXT Col
NEXT Row

// Searching a 1-D array (linear search)
Found ← FALSE
FOR i ← 1 TO 30
   IF Marks[i] = Target THEN
      Found ← TRUE
      Position ← i
   ENDIF
NEXT i

Watch out Using Marks[0] is out-of-bounds for ARRAY[1:30] — start at 1.

6. Procedures (no return value)

A procedure performs an action. Parameters are typed. Call with CALL and the procedure name.

// Procedure with one parameter
PROCEDURE PrintHello(Name : STRING)
   OUTPUT "Hello, ", Name
ENDPROCEDURE

// Calling it
CALL PrintHello("Alex")
CALL PrintHello("Sam")

// Procedure with no parameters
PROCEDURE DrawLine
   OUTPUT "----------"
ENDPROCEDURE

CALL DrawLine

// Procedure with multiple parameters
PROCEDURE PrintBill(Item : STRING, Price : REAL)
   OUTPUT Item, " costs $", Price
ENDPROCEDURE

CALL PrintBill("Coffee", 3.50)
CALL PrintBill("Cake", 4.25)

Key idea Procedures use CALL. Functions don't — they appear inside expressions.

7. Functions (returns a value)

A function computes and returns a value. Use it inside expressions or assignments — never with CALL.

// Function returning an INTEGER
FUNCTION Square(N : INTEGER) RETURNS INTEGER
   RETURN N * N
ENDFUNCTION

// Using it in an assignment
Answer ← Square(7)        // Answer = 49

// Using it inside an expression
Total ← Square(3) + Square(4)   // 9 + 16 = 25

// Using it inside an OUTPUT
OUTPUT "7 squared = ", Square(7)

// Function with REAL parameter and return
FUNCTION CircleArea(R : REAL) RETURNS REAL
   RETURN 3.14159 * R * R
ENDFUNCTION

Area ← CircleArea(5.0)

// Function returning BOOLEAN
FUNCTION IsEven(N : INTEGER) RETURNS BOOLEAN
   IF N MOD 2 = 0 THEN
      RETURN TRUE
   ELSE
      RETURN FALSE
   ENDIF
ENDFUNCTION

IF IsEven(Number) THEN
   OUTPUT "Even"
ENDIF

Common mistake Writing CALL Square(7) — wrong, it's a function so just use Square(7).

8. String Operations

Cambridge string functions are 1-indexed. Memorise the four below and the & concatenation operator.

// LENGTH — number of characters
DECLARE Word : STRING
Word ← "Computer"
N ← LENGTH(Word)          // N = 8

// LCASE — convert to lowercase
Lower ← LCASE("HELLO")     // "hello"

// UCASE — convert to uppercase
Upper ← UCASE("hello")     // "HELLO"

// SUBSTRING(s, start, length) — 1-indexed
Word ← "Computer"
Part ← SUBSTRING(Word, 1, 3)   // "Com"
Mid  ← SUBSTRING(Word, 4, 3)   // "put"
End  ← SUBSTRING(Word, 7, 2)   // "er"

// Concatenation with &
DECLARE FirstName : STRING
DECLARE LastName  : STRING
FirstName ← "Alex"
LastName  ← "Smith"
FullName ← FirstName & " " & LastName
// FullName = "Alex Smith"

// Worked example: extract first 3 chars
INPUT Word
IF LENGTH(Word) >= 3 THEN
   Code ← UCASE(SUBSTRING(Word, 1, 3))
   OUTPUT "Code: ", Code
ELSE
   OUTPUT "Word too short"
ENDIF

Indexing SUBSTRING("Hello", 1, 1) = "H" — the first character is at position 1, not 0.

9. Operators

Three families: arithmetic, comparison, logical. DIV and MOD always trip students up — see the table.

// Arithmetic: + - * / DIV MOD
A ← 7 + 2            // 9
B ← 7 - 2            // 5
C ← 7 * 2            // 14
D ← 7 / 2            // 3.5  (REAL division)

// DIV — integer division (drops remainder)
E ← 7 DIV 2          // 3
F ← 15 DIV 4         // 3
G ← 100 DIV 3        // 33

// MOD — remainder of integer division
H ← 7 MOD 2          // 1

// Comparison: returns BOOLEAN
//   =   equal
//   <>  not equal
//   <   less than
//   >   greater than
//   <=  less than or equal
//   >=  greater than or equal

IF Name <> "Alex" THEN ...

// Logical: AND, OR, NOT

   OUTPUT "Weekend"

IF NOT Found THEN
   OUTPUT "Item missing"
ENDIF

Useful pattern N MOD 2 = 0 tests for even; N MOD 10 gives the last digit.

10. File Handling

Files must be opened in a mode, used, then closed. The three modes are READ, WRITE, APPEND.

// Open a file in one of three modes
OPENFILE "data.txt" FOR READ      // read existing
OPENFILE "data.txt" FOR WRITE     // overwrite/create
OPENFILE "data.txt" FOR APPEND    // add to end

// Read one line into a variable
READFILE "data.txt", LineVariable

// Write a line
WRITEFILE "data.txt", "text to write"

// Always close when done
CLOSEFILE "data.txt"

// EOF() returns TRUE when at end of file
// Full read-and-print pattern:
OPENFILE "names.txt" FOR READ
WHILE NOT EOF("names.txt") DO
   READFILE "names.txt", Line
   OUTPUT Line
ENDWHILE
CLOSEFILE "names.txt"

// Writing several lines
OPENFILE "log.txt" FOR WRITE
FOR i ← 1 TO 5
   WRITEFILE "log.txt", "Entry " & i
NEXT i
CLOSEFILE "log.txt"

// Append to an existing log
OPENFILE "log.txt" FOR APPEND
WRITEFILE "log.txt", "new event"
CLOSEFILE "log.txt"

Don't forget CLOSEFILE — easy mark lost in markschemes.

11. Data Types Deep Dive

Six built-in types. Watch the quotes: single for CHAR, double for STRING.

// INTEGER — whole numbers (positive, negative, zero)
DECLARE Score : INTEGER
Score ← 42
Score ← -7

// REAL — decimal/fractional numbers
DECLARE Temp : REAL
Temp ← 36.6
Temp ← -2.5
Temp ← 3.0      // still REAL, not INTEGER

// CHAR — exactly ONE character, single quotes
DECLARE Grade : CHAR
Grade ← 'A'
Initial ← 'Z'
Digit ← '7'     // the character '7', not the number

// STRING — text, double quotes
DECLARE Name : STRING
Name ← "Alex"
Message ← "Hello, world!"
Empty ← ""          // empty string is fine

// BOOLEAN — TRUE or FALSE only
DECLARE Active : BOOLEAN
Active ← TRUE
Active ← FALSE

// DATE — dd/mm/yyyy
DECLARE Today : DATE
Today ← 19/05/2026
Birthday ← 01/01/2010

Quote rule 'A' is CHAR. "A" is STRING. They're not the same type.

12. Operator Precedence

When a line mixes operators, evaluate in this order (highest first):

  1. ( ) — parentheses
  2. ^ — exponentiation (if used)
  3. *, /, DIV, MOD
  4. +, -
  5. = <> < > <= >= — comparison
  6. NOT
  7. AND
  8. OR
// Worked examples — left-to-right within a level
5 + 3 * 2             // = 5 + 6     = 11
(5 + 3) * 2           // = 8 * 2     = 16
20 - 6 / 2            // = 20 - 3    = 17
10 DIV 3 + 1          // = 3 + 1     = 4
10 MOD 3 * 2          // = 1 * 2     = 2

// Comparison + logical
A > 5 AND B < 10     // compare first, then AND
NOT X = Y               // equals first, then NOT

// Use brackets to make intent explicit
IF (Age >= 13) AND (Age <= 19) THEN
   OUTPUT "Teen"
ENDIF

Safe rule If in doubt, add brackets. Examiners reward clarity.

Bonus: Comments & Layout

Cambridge accepts both inline comments and well-formatted indentation. Add comments to earn structure marks.

// This is a single-line comment

How to draw a flowchart

Exam method
  • · Always begin with START and end with STOP — both in ovals.
  • · Use arrows to show direction. Default flow is top → bottom, left → right; loops curve back upwards.
  • · Every decision diamond splits into exactly two paths, labelled Yes/No or True/False.
  • · Inputs and outputs always use the parallelogram. Don't put them in a rectangle.
  • · Calculations and assignments use the rectangle: Total ← Total + N.
  • · Keep symbols roughly the same size and aligned on a grid — examiners mark for clarity.
  • · Plan on paper first; rub out before committing — every line should land at one shape.
  • · Use connectors (A, B…) only if your diagram is large and arrows would cross the page.

How to read a flowchart

Trace method
  • · Treat it like a trace table — start at START and follow the arrows.
  • · When you reach a decision diamond, evaluate the condition with current values and follow the matching label.
  • · Each process box updates a variable — write the new value next to its column in your trace.
  • · Each I/O parallelogram: take the next input value from the question, or record the output in your OUTPUT column.
  • · Loops repeat the arrows back upwards — keep going around until the decision condition fails.
  • · Stop only when you reach STOP. Then list the final outputs in order.
Tip: mark each arrow you've travelled with a tick — it stops you re-entering a loop body in the wrong place.

Searching Algorithms

Two methods you must know cold: Linear and Binary. Know the pseudocode, the trade-offs, and how to trace each by hand.

Linear Search

Sequential

Check each element from start to end, one at a time, until the target is found or the list ends.

  • Works on unsorted data — major advantage
  • Worst case: up to n comparisons (target last or absent)
  • Best case: 1 comparison (target first)
  • Time complexity (informal): linear — doubles list, doubles time
Pseudocode
Found ← False
Index ← 1
WHILE Index <= n AND Found = False DO
  IF List[Index] = Target THEN
    Found ← True
    Position ← Index
  ENDIF
  Index ← Index + 1
ENDWHILE
IF Found = True THEN
  OUTPUT "Found at position ", Position
ELSE
  OUTPUT "Not found"
ENDIF
Python equivalent
def linear_search(lst, target):
    for i, value in enumerate(lst):
        if value == target:
            return i  # index found
    return -1      # not found

Binary Search

Divide & Conquer

List MUST be sorted. Look at the middle item; eliminate half the list each step.

  • Precondition: data must be in order
  • Time complexity: O(log n) — much faster for large lists
  • 1024 items → at most 10 comparisons
  • 1,000,000 items → at most 20 comparisons
Pseudocode
Low ← 1
High ← n
Found ← False
WHILE Low <= High AND Found = False DO
  Mid ← (Low + High) DIV 2
  IF List[Mid] = Target THEN
    Found ← True
    Position ← Mid
  ELSE
    IF List[Mid] < Target THEN
      Low ← Mid + 1
    ELSE
      High ← Mid - 1
    ENDIF
  ENDIF
ENDWHILE
Python equivalent
def binary_search(lst, target):
    low, high = 0, len(lst) - 1
    while low <= high:
        mid = (low + high) // 2
        if lst[mid] == target:
            return mid
        elif lst[mid] < target:
            low = mid + 1
        else:
            high = mid - 1
    return -1

Model answer:

Q3 — State two advantages and one disadvantage of binary search compared to linear search. 3 marks

Advantages (any 2):

  • Faster on large lists — eliminates half each step (O(log n) vs O(n)).
  • Fewer comparisons on average and in the worst case.
  • More efficient when the same list is searched repeatedly.

Disadvantage (any 1):

  • Requires the data to be sorted first — sorting adds extra cost.
  • More complex to code (Low/High/Mid logic) and harder to debug.
  • For very small lists or unsorted data, linear is just as fast or faster overall.

Sorting — Bubble Sort

The only sort you must know for IGCSE. Get the nested loop, the swap with Temp, and the Swapped flag for early termination locked in.

How it works

  1. Go through the list comparing each adjacent pair.
  2. If a pair is out of order, swap them (using a Temp variable).
  3. After one full pass, the largest remaining unsorted value has "bubbled" to the end.
  4. Repeat: each subsequent pass needs one fewer comparison (the tail is already sorted).
  5. If a pass completes with no swaps, the list is sorted — exit early.

Why "bubble"? Large values rise to the top end of the list like air bubbles in water.

Pseudocode (with Swapped flag)
FOR i ← 1 TO n - 1
  Swapped ← False
  FOR j ← 1 TO n - i
    IF List[j] > List[j+1] THEN
      Temp ← List[j]
      List[j] ← List[j+1]
      List[j+1] ← Temp
      Swapped ← True
    ENDIF
  NEXT j
  IF Swapped = False THEN
    // Early termination — list is sorted
    EXIT FOR
  ENDIF
NEXT i

Note: n - i ensures we don't recompare items already in their final position.

Python equivalent
def bubble_sort(lst):
    n = len(lst)
    for i in range(n - 1):
        swapped = False
        for j in range(n - 1 - i):
            if lst[j] > lst[j + 1]:
                # tuple swap — no Temp needed
                lst[j], lst[j + 1] = lst[j + 1], lst[j]
                swapped = True
        if not swapped:
            break  # early exit
    return lst

# Example
print(bubble_sort([5, 2, 4, 1]))
# → [1, 2, 4, 5]

Standard Methods of Solution

Five tiny patterns you must be able to write from memory: Totalling Counting Max Min Average. Most Paper 2 questions combine 2–3 of these.

Totalling (Sum)

Σ

Add every input value to a running total. Always initialise Total to 0 before the loop.

Total ← 0
FOR i ← 1 TO 10
  INPUT Num
  Total ← Total + Num
NEXT i
OUTPUT Total

Counting (Conditional)

#

Increment a counter only when a condition is met. Initialise Count to 0.

Count ← 0
FOR i ← 1 TO 10
  INPUT Num
  IF Num > 50 THEN
    Count ← Count + 1
  ENDIF
NEXT i
OUTPUT Count

Finding Maximum

↑

Initialise Max to the first value, then compare every later value.

INPUT Max          // first value
FOR i ← 2 TO 10
  INPUT Num
  IF Num > Max THEN
    Max ← Num
  ENDIF
NEXT i
OUTPUT Max

Finding Minimum

↓

Same structure as Max — but compare with < Min.

INPUT Min          // first value
FOR i ← 2 TO 10
  INPUT Num
  IF Num < Min THEN
    Min ← Num
  ENDIF
NEXT i
OUTPUT Min

Finding the Average

Combine totalling with division by the count of inputs at the end. Be careful: divide after the loop, not inside it.

Total ← 0
FOR i ← 1 TO 10
  INPUT Num
  Total ← Total + Num
NEXT i
Average ← Total / 10
OUTPUT Average

Tip If the number of values is variable, count them in a separate counter and divide by that — never hard-code the divisor.

Combined: Max, Min & Average in One Pass

A typical Paper 2 question asks for multiple results from a single loop. Pattern: read first value → set Max and Min to it → loop the remaining values.

Total ← 0

// Read first value and use it to seed Max and Min
INPUT Num
Max ← Num
Min ← Num
Total ← Total + Num

FOR i ← 2 TO 10
  INPUT Num
  Total ← Total + Num
  IF Num > Max THEN
    Max ← Num
  ENDIF
  IF Num < Min THEN
    Min ← Num
  ENDIF
NEXT i

Average ← Total / 10

OUTPUT "Maximum: ", Max
OUTPUT "Minimum: ", Min
OUTPUT "Average: ", Average

Notice we only loop from 2 to 10 — the first value was already consumed before the loop. This is the safest pattern.

Important Gotcha

DANGER

Don't initialise Max to 0 (or Min to a huge number) — initialise to the FIRST value!

  • If you set Max ← 0 and all inputs are negative (e.g. temperatures -12, -5, -8), the algorithm reports 0 — which is not in the list and is wrong.
  • If you set Min ← 999 and all inputs are larger (e.g. 1200, 1500, 1100), the algorithm reports 999 — again not even a real input.
  • Fix: read the first input before the loop and assign it to both Max and Min. Then loop from index 2 onwards.
Wrong
Max ← 0
FOR i ← 1 TO 10
  INPUT Num
  IF Num > Max THEN
    Max ← Num
  ENDIF
NEXT i  // Breaks for all-negative input
Correct
INPUT Max
FOR i ← 2 TO 10
  INPUT Num
  IF Num > Max THEN
    Max ← Num
  ENDIF
NEXT i  // Robust

Three traps that cost marks

  1. Forgetting to cast input(). Writing age = input() then if age > 18: raises TypeError: '>' not supported between instances of 'str' and 'int'. Fix: age = int(input()).
  2. Off-by-one in range(). range(1, 10) produces 1, 2, ..., 9 — it does NOT include 10. For pseudocode FOR i ← 1 TO 10 write range(1, 11).

More traps (bonus)

  1. Confusing = (assign) with == (compare). Pseudocode uses = for both!
  2. Forgetting the : at the end of if / for / while / def lines.
  3. Indentation matters in Python — it defines the block. Mixing tabs and spaces causes IndentationError.
  4. print(a, b) puts a space between values; pseudocode OUTPUT a, b concatenates with no separator.

Procedures and functions are named, reusable blocks of code. They help with decomposition — breaking a big problem into smaller, testable pieces.

Parameters, arguments, and scope

  • Parameter — the variable in the function/procedure definition. Example: Width in PROCEDURE PrintRow(Width : INTEGER), or n in def square(n):.
  • Argument — the actual value passed in when the procedure or function is called. Example: the 5 in CALL PrintRow(5), or the 7 in square(7).
  • Local variable — declared inside the function/procedure. Only exists while that block is running; it cannot be seen from outside.
  • Global variable — declared at the top level of the program; visible everywhere. In Python you can read a global inside a function, but to modify it you must write global name inside the function first.

Why use procedures and functions?

  • Code reuse — write once, call many times.
  • Easier to test individual pieces — supports decomposition.
  • More readable — a meaningful name describes a block of logic at a glance.
  • Less duplication — one place to fix a bug instead of many.
  • Easier maintenance — changes are contained; less risk of breaking unrelated code.
  • Team work — different programmers can write different procedures in parallel.

Multiple parameters & multiple return values

Pseudocode
FUNCTION MinMax(A : ARRAY[1:10] OF INTEGER)
        RETURNS INTEGER
  Lo ← A[1]
  Hi ← A[1]
  FOR i ← 2 TO 10
    IF A[i] < Lo THEN Lo ← A[i] ENDIF
    IF A[i] > Hi THEN Hi ← A[i] ENDIF
  NEXT i
  RETURN Hi - Lo          // range
ENDFUNCTION
Python (returns a tuple of both)
def min_max(a):
    lo = a[0]
    hi = a[0]
    for n in a[1:]:
        if n < lo: lo = n
        if n > hi: hi = n
    return lo, hi      # tuple

low, high = min_max([3, 7, 1, 9, 4])
print(low, high, high - low)

Files give programs persistent storage — data survives after the program ends. For Paper 2 you need to read, write, and append to plain text files in both pseudocode and Python. Always remember to open → use → close.

I/O Text files

Reading a file

Idiomatic Python uses a with block (with open("scores.txt") as f: ...) which auto-closes the file even on error. IGCSE mark schemes accept either, but typically expect explicit open and close.

Writing to a file

Pseudocode
OPENFILE "log.txt" FOR WRITE
WRITEFILE "log.txt", "Run complete"
CLOSEFILE "log.txt"
Python
f = open("log.txt", "w")
f.write("Run complete\n")   # add \n yourself!

Opening with "w" overwrites any existing file with the same name. Be careful — data already in the file is lost.

Appending

Pseudocode
OPENFILE "log.txt" FOR APPEND
WRITEFILE "log.txt", "Another entry"
CLOSEFILE "log.txt"
Python
f = open("log.txt", "a")   # a = append
f.write("Another entry\n")
f.close()

"a" mode adds to the end of the file and creates it if missing — existing data is kept.

Worked example: sum numbers from a file

File data.txt has one integer per line. Read them all and output the total.

Pseudocode

  READFILE "data.txt", Num

ENDWHILE

Common file errors

  • Forgetting to close the file. Data written may stay in a buffer and never reach disk. Always f.close() (or use a with block).
  • Reading a missing file. open("missing.txt", "r") raises FileNotFoundError. Check existence first or wrap in try / except.
  • Forgetting \n when writing. Python's write() does not auto-append a newline — without it everything ends up on one line.
  • Reading numbers without casting. line is a string. "5" + 3 raises TypeError. Use int(line.strip()) or float(line.strip()).
  • Opening with "w" when you meant "a". "w" wipes existing content — a frequent and painful mistake.

Databases & SQL

In IGCSE 0478/0984 you write single-table SQL queries using SELECT, FROM, WHERE, ORDER BY, the aggregate functions and (in the 2026 syllabus) GROUP BY — applied to a flat table given in the question.

Database Terminology

Vocab

Examiners deduct marks for sloppy use of "row" when you mean "record", or "field" when you mean "data type". Learn these precisely.

Table

A structured collection of data, organised in rows and columns. Each table stores information about one type of entity (e.g. PRODUCT, STUDENT, BOOK).

Record (row)

A single entry in a table — one entity, e.g. one product. All the values for one item, read left to right.

Field (column)

A single attribute — one property of an entity, e.g. Price. Every record has a value for every field (possibly NULL).

Primary key

A field (or combination of fields) that uniquely identifies each record. Two rules: no duplicates and never NULL. Example: ProductID.

Foreign key

A field in one table that references the primary key of another table, linking the two. IGCSE 2026 is mostly single-table, but the vocabulary is examinable.

Database

A collection of related tables, organised so that data can be efficiently stored, retrieved and updated. Managed by software called a DBMS.

Tip: in exam answers always say "record" not "row" and "field" not "column" — those are the marker's keywords.

Basic SELECT

4 — Simple WHERE

SELECT * FROM PRODUCT
WHERE Category = "Electronics";
What this returns

4 records: P001 Laptop X1, P003 Bluetooth Headphones, P006 Smartphone Z9, P009 USB Cable.

SELECT Name FROM PRODUCT
WHERE Stock <> 0;

9 names — every product except Bluetooth Headphones (the only one whose Stock = 0).

Combining conditions — AND, OR

7 — AND
AND means both conditions must be true.
SELECT * FROM PRODUCT
WHERE Category = "Grocery"
  AND Price < 10;
What this returns

2 records: P005 Tea Bags 100ct (4.99) and P008 Olive Oil 500ml (8.99). Coffee Beans is Grocery but costs 12.50 so it's excluded.

8 — OR
OR means at least one condition is true.
SELECT * FROM PRODUCT
WHERE Category = "Electronics"
   OR Category = "Furniture";
What this returns

7 records — every product except the 3 Grocery items (Coffee Beans, Tea Bags, Olive Oil).

9 — AND + OR with parentheses
Mix AND/OR carefully — use parentheses to force the evaluation order. Without them, AND binds tighter than OR and your query may filter the wrong rows.
SELECT * FROM PRODUCT
WHERE Available = TRUE
  AND (Price < 50 OR Stock > 100);
What this returns

Available items that are either cheap (< 50) or abundantly stocked (> 100): P002 Coffee Beans (price 12.50), P005 Tea Bags (price 4.99 and stock 120), P007 Desk Lamp (39.99), P008 Olive Oil (8.99), P009 USB Cable (5.99, stock 200) — 5 records.

10 — LIKE with %
% matches any number of characters (including zero).
SELECT Name FROM PRODUCT
WHERE Name LIKE "S%";
-- names starting with capital S
What this returns

2 rows: Smartphone Z9, Standing Desk.

11 — LIKE with _
_ matches exactly one character. The pattern "P00_" is 4 characters: P, 0, 0, then any single character.
SELECT * FROM PRODUCT
WHERE ProductID LIKE "P00_";
What this returns

9 records — P001 through P009. P010 is excluded because its ID is 4 characters but the last is not a single-position match (the IDs are length 4, so this actually returns all 10? SELECT * FROM PRODUCT
WHERE Name LIKE "%oil%";
What this returns

1 record: Olive Oil 500ml — the only name containing the letters "oil".

13 — BETWEEN
BETWEEN x AND y is shorthand for >= x AND <= y — inclusive at both ends.
SELECT Name, Price FROM PRODUCT

What this returns

4 rows: Coffee Beans 1kg / 12.50, Desk Lamp / 39.99, Olive Oil 500ml / 8.99, USB Cable / 5.99. (Tea Bags at 4.99 is just below 5, so excluded.)

SELECT * FROM PRODUCT

4 records: P001 Laptop X1, P002 Coffee Beans, P005 Tea Bags, P006 Smartphone Z9.

SELECT Name, Price FROM PRODUCT

10 rows in price order: Tea Bags / 4.99, USB Cable / 5.99, Olive Oil / 8.99, Coffee Beans / 12.50, Desk Lamp / 39.99, Bluetooth Headphones / 79.00, Office Chair / 249.99, Standing Desk / 449.00, Smartphone Z9 / 599.00, Laptop X1 / 899.99.

SELECT COUNT(*) FROM PRODUCT

Schema commands

SQL is split into two main parts:

  • DDL — Data Definition Language: defines and changes the structure of tables. Commands: CREATE TABLE, ALTER TABLE, DROP TABLE.
  • DML — Data Manipulation Language: works with the data in tables. Commands: SELECT, INSERT, UPDATE, DELETE (this is what we've been doing).
CREATE TABLE

Defines a new table: name, fields, data types, and constraints such as PRIMARY KEY and NOT NULL.

ALTER TABLE

Modifies an existing table — typically ADD a new field, change a data type, or drop a column.

DROP TABLE

Permanently deletes the table and all its data. Irreversible — examiners love asking why this is dangerous.

Example — creating the PRODUCT table

CREATE TABLE PRODUCT (
  ProductID TEXT PRIMARY KEY,
  Name TEXT NOT NULL,
  Category TEXT,
  Price REAL,
  Stock INTEGER,
  Available BOOLEAN
);

Notes: each field is name TYPE [constraints], separated by commas; PRIMARY KEY follows the type of the field that is the key; NOT NULL forces a value to be supplied; a semicolon ends the statement.

Common SQL Mistakes

Lose-mark traps
MistakeWrongCorrect
Forgetting quotes around text values — every year examiners report this as the most common error. WHERE Category = Electronics WHERE Category = "Electronics"
Using == for equality (a habit from Python/Java). SQL uses a single =. WHERE Price == 50 WHERE Price = 50
Using != instead of <>. <> is the standard, exam-safe operator; != works in some dialects only. WHERE Stock != 0 WHERE Stock <> 0
Clauses in the wrong order — ORDER BY must come after WHERE. SELECT * FROM PRODUCT ORDER BY Price WHERE Stock > 0; SELECT * FROM PRODUCT WHERE Stock > 0 ORDER BY Price;
Missing semicolon at the end of a query — some markschemes accept this, but the safe habit is to always include it. SELECT * FROM PRODUCT SELECT * FROM PRODUCT;
Putting an aggregate inside WHERE — illegal. Aggregates go in SELECT (or filtered with HAVING after GROUP BY). WHERE SUM(Stock) > 100 SELECT SUM(Stock) FROM ... GROUP BY ...;
Quoting numbers — numbers are never quoted; quotes turn them into strings, breaking comparisons. WHERE Price < "50" WHERE Price < 50
Using LIKE without wildcards — works, but is exactly equivalent to =, so pointless. WHERE Name LIKE "Laptop X1" WHERE Name LIKE "Laptop%" (or use =)

Practice — full question

  1. (a) Output the names of all products priced above £100.
  2. (b) Output the total stock value (Price × Stock summed) for all available products.
  3. (c) Output the supplier with the highest total stock. (Use GROUP BY.)
  4. (d) Output all products in the "Electronics" category, sorted by price from highest to lowest.
  5. (e) Output the average price of products in each category.
  6. (f) Output all products whose name contains the word "Bean" or "Coffee".
Model answers (click to reveal)

(a) Names of products priced above £100 — 2 marks

SELECT Name
FROM PRODUCT
WHERE Price > 100;

Marks: 1 for SELECT Name FROM PRODUCT; 1 for the correct WHERE Price > 100. Result: Laptop X1, Office Chair, Smartphone Z9, Standing Desk (4 rows).

(b) Total stock value of available products — 2 marks

SELECT SUM(Price * Stock)
FROM PRODUCT
WHERE Available = TRUE;

Marks: 1 for the SUM(Price * Stock) aggregate; 1 for the WHERE Available = TRUE filter. Excludes P003 and P010 which are FALSE.

(c) Supplier with the highest total stock — 2 marks

SELECT Supplier, SUM(Stock)
FROM PRODUCT
GROUP BY Supplier
ORDER BY SUM(Stock) DESC;

Marks: 1 for the GROUP BY Supplier; 1 for the ORDER BY SUM(Stock) DESC. The first row of the output is the answer — AudioPlus, with 200 units (Bluetooth Headphones 0 + USB Cable 200). Showing the first row only would normally be done with LIMIT 1, but IGCSE does not require LIMIT — the ordered list is accepted.

(d) Electronics products sorted by price descending — 2 marks

SELECT *
FROM PRODUCT
WHERE Category = "Electronics"
ORDER BY Price DESC;

Marks: 1 for the WHERE Category = "Electronics"; 1 for ORDER BY Price DESC. Result order: Laptop X1 (899.99), Smartphone Z9 (599.00), Bluetooth Headphones (79.00), USB Cable (5.99).

(e) Average price per category — 2 marks

SELECT Category, AVG(Price)
FROM PRODUCT
GROUP BY Category;

Marks: 1 for the AVG(Price) aggregate selected alongside Category; 1 for the GROUP BY Category. Result: Electronics ≈ 395.99, Furniture ≈ 246.33, Grocery ≈ 8.83.

(f) Products whose name contains "Bean" or "Coffee" — 2 marks

SELECT *
FROM PRODUCT
WHERE Name LIKE "%Bean%"
   OR Name LIKE "%Coffee%";

Marks: 1 for using LIKE with the %...% wildcards on both sides; 1 for combining the two patterns with OR. Result: P002 Coffee Beans 1kg (matches both patterns — returned once).

Identify Errors in Code

A code-with-errors question is virtually guaranteed on Paper 2. You will be given a pseudocode or Python snippet containing 2–4 bugs and asked to identify the line, explain the bug, and write a corrected line. Work through these 10 problems — if you can fix all of them, you are exam-ready for this style of question.

Problem 1 — find the error(s)

Pseudocode that should sum 10 numbers and output the total.

// Sum 10 numbers
Total ← 0
FOR i ← 1 TO 10
  INPUT Num
  Total = Total + Num
NEXT i
OUTPUT Total
Show answer

Bug: Line 5 uses = for assignment. In pseudocode, = is only for comparison (equality). Assignment is always ←. The examiner will dock you a mark every time you do this.

Corrected line:

  Total ← Total + Num
Problem 2 — find the error(s)

Pseudocode that should find the largest of 5 input numbers (which may be negative).

Max ← 0
FOR i ← 1 TO 5
  INPUT Num
  IF Num > Max THEN
    Max ← Num
  ENDIF
NEXT i
OUTPUT Max
Show answer

Bug: Max is initialised to 0. If every input is negative (e.g. −5, −9, −2, −7, −1) then 0 stays as the maximum — but 0 was never actually input. The "max so far" must be a real input.

Fix: read the first number outside the loop, set Max to it, then loop 4 more times:

INPUT Num
Max ← Num
FOR i ← 2 TO 5
  INPUT Num
  IF Num > Max THEN Max ← Num
NEXT i
Problem 3 — find the error(s)

Python that should average 10 marks.

total = 0
for i in range(10):
    mark = input("Mark: ")
    total = total + mark
print(total / 10)
Show answer

Bug: input() always returns a string. Adding a string to the integer total raises TypeError: unsupported operand type(s) for +: 'int' and 'str'. You must cast to int.

Corrected line:

    mark = int(input("Mark: "))
Problem 4 — find the error(s)

A Python bubble sort attempt.

nums = [5, 2, 8, 1, 9]
for i in range(len(nums)):
    for j in range(len(nums)):
        if nums[j] > nums[j+1]:
            nums[j], nums[j+1] = nums[j+1], nums[j]
Show answer

Bug: When j = len(nums)-1 = 4, the line nums[j+1] accesses index 5 — out of range — raising IndexError. The inner loop must stop one short.

Fix:

    for j in range(len(nums) - 1 - i):

Subtracting i is also an efficiency improvement (the last i items are already sorted on each pass).

Problem 5 — find the error(s)

Pseudocode declaring and filling an array of 5 integers.

DECLARE Arr : ARRAY[1:5] OF INTEGER
FOR i ← 0 TO 5
  INPUT Arr[i]
NEXT i
Show answer

Two bugs:

  1. Pseudocode arrays declared as ARRAY[1:5] are 1-indexed. Arr[0] does not exist — accessing it is an error.
  2. The loop runs 0,1,2,3,4,5 — that is 6 iterations, but the array only has 5 slots.

Fix:

FOR i ← 1 TO 5
Problem 6 — find the error(s)

Pseudocode that should ask for a password and keep asking until the user gets it right.

REPEAT
  INPUT Password
UNTIL Password <> "secret"
Show answer

Bug: UNTIL in pseudocode means "loop ends when this condition becomes TRUE". The condition Password <> "secret" is true when the password is wrong, so the loop exits on the first wrong guess — the opposite of what was wanted.

Fix:

UNTIL Password = "secret"

Remember: REPEAT … UNTIL <exit condition>. The condition is what makes the loop stop, not continue.

Problem 7 — find the error(s)

Python that should sum the integers 1 to 10 inclusive (answer should be 55).

total = 0
for i in range(1, 10):
    total = total + i
print(total)  # prints 45, expected 55
Show answer

Bug: range(a, b) in Python is exclusive of b. range(1, 10) gives 1, 2, …, 9 — sum is 45. To include 10, the upper bound must be 11.

Fix:

for i in range(1, 11):

Contrast with pseudocode: FOR i ← 1 TO 10 is inclusive of both ends.

Problem 8 — find the error(s)

Pseudocode that should output the numbers 1 to 5.

FOR i ← 1 TO 5
  OUTPUT i
ENDIF
Show answer

Bug: The loop is closed with ENDIF. That keyword closes an IF … THEN block. A FOR loop closes with NEXT i.

Fix:

NEXT i

Quick reference of closers: IF → ENDIF, FOR → NEXT i, WHILE → ENDWHILE, REPEAT → UNTIL, CASE → ENDCASE, PROCEDURE → ENDPROCEDURE, FUNCTION → ENDFUNCTION.

Problem 9 — find the error(s)

An SQL query that should return all electronics products.

SELECT * FROM PRODUCT WHERE Category = Electronics;
Show answer

Bug: Electronics with no quotes is interpreted by SQL as a column name, not a string literal. Either the query fails (no such column) or it does something completely unintended.

Fix: wrap text values in quotes.

SELECT * FROM PRODUCT WHERE Category = "Electronics";

Numbers (Price > 50) and booleans (InStock = TRUE) are not quoted. Only text and dates.

Problem 10 — find the error(s)

Pseudocode that validates that an age is between 0 and 120 inclusive.

INPUT Age
IF Age < 0 OR Age > 120 THEN
  OUTPUT "Valid"
ELSE
  OUTPUT "Invalid"
ENDIF
Show answer

Bug: The condition Age < 0 OR Age > 120 is true when the age is out of range — i.e. invalid. The outputs are swapped.

Fix: swap the outputs, or invert the condition:

IF Age >= 0 AND Age <= 120 THEN
  OUTPUT "Valid"
ELSE
  OUTPUT "Invalid"
ENDIF

Note: an age range check is range validation, not length or type. State the right validation name in 1-mark questions.

Common Mistakes Examiners Penalise

These are the recurring deductions seen in every examiner report. Most of them lose only 1 mark each — but a paper with five of these slips is the difference between an A and an A*. Read all of them tonight.

Mistake 1

Using = for assignment in pseudocode

Pseudocode uses ← for assignment and = only for comparison. Writing x = 5 in pseudocode is wrong even if "everyone knows what you mean".

// Wrong
Count = Count + 1
// Right
Count ← Count + 1
Mistake 2

Forgetting to initialise Total or Count before the loop

Adding to a variable that was never set is undefined behaviour. Always set running totals and counters to 0 (or 1) before the loop begins.

// Wrong — Total used before being set
FOR i ← 1 TO 10
  Total ← Total + i
NEXT i

// Right
Total ← 0
FOR i ← 1 TO 10
  Total ← Total + i
NEXT i
Mistake 3

Off-by-one in FOR i ← 1 TO 9

Pseudocode TO is inclusive on both ends. FOR i ← 1 TO 10 is 10 iterations, FOR i ← 1 TO 9 is only 9. Python range(1, 10) is also only 9. Always check whether the question wanted "the first 10" or "1 to 10".

Mistake 4

Not casting int(input()) in Python

A bare input() returns a string. If you intend to do arithmetic or numeric comparison, you must cast.

# Wrong
n = input("Enter: ")
if n > 10: ...   # TypeError

# Right
n = int(input("Enter: "))
Mistake 5

Omitting validation when the question explicitly asks for it

If the question says "the program must validate the input is between 1 and 100", you must include an IF (or REPEAT … UNTIL) check — not just accept input. State the validation type in commentary where possible (range, length, type, presence, format, lookup/check digit).

Mistake 6

Forgetting quotes around SQL text values

Text and dates need quotes; numbers and booleans don't. Single or double quotes both work, but be consistent within one query.

-- Wrong
SELECT * FROM Student WHERE Name = Smith;
-- Right
SELECT * FROM Student WHERE Name = "Smith";
Mistake 7

Trace tables: one row per line instead of per change

A trace table records the value of each variable when it changes. You do not write a new row for OUTPUT or for an IF evaluation that doesn't update anything. Output usually has its own column.

Mistake 8

Confusing validation and verification

Validation = is the data sensible? (range, length, type, presence, format, lookup) — done by the computer. Verification = was the data copied/entered correctly? (double-entry, visual check) — about transcription, not sense.

Mistake 9

Bubble sort without a Swapped flag

The full mark scheme version of bubble sort uses a Swapped boolean so the algorithm can exit early. Forgetting it usually still scores most marks — but if the question says "most efficient" or "stop when sorted", you must include it.

REPEAT
  Swapped ← FALSE
  FOR i ← 1 TO n - 1
    IF A[i] > A[i+1] THEN
      // swap
      Swapped ← TRUE
    ENDIF
  NEXT i
UNTIL Swapped = FALSE
Mistake 10

Returning the value when asked for the position

For linear/binary search questions, read whether the output should be the item found, its index/position, or just a boolean "yes/no found". The three are different answers.

Mistake 11

Forgetting CLOSEFILE / f.close()

Any file question expects the file to be closed at the end. It's a free mark — never skip it.

OPENFILE "data.txt" FOR READ
  READFILE "data.txt", Line
CLOSEFILE "data.txt"
Mistake 12

Writing Python elif in pseudocode

Pseudocode does not have elif. Use nested IF … ELSE IF, or much better, CASE OF … ENDCASE for multi-way selection.

// Multi-way preferred form
CASE OF Grade
  "A" : OUTPUT "Excellent"
  "B" : OUTPUT "Good"
  "C" : OUTPUT "Pass"
  OTHERWISE : OUTPUT "Fail"
ENDCASE
Mistake 13

Initialising Max with 0 (fails for negative inputs)

The safe pattern is to initialise Max with the first input value, then loop over the rest. The reverse mistake is initialising Min with 0 when inputs may be positive.

Mistake 14

Misreading <= as < in range checks

"Between 1 and 10 inclusive" means >= 1 AND <= 10, not > 0 AND < 11 (which is the same, but harder to read), and definitely not > 1 AND < 10 (which excludes 1 and 10). Always re-check the boundary.

Mistake 15

Pseudocode arrays treated as 0-indexed

A pseudocode array declared ARRAY[1:n] starts at index 1, not 0. Loops should start at 1. Confusing Python's 0-indexed lists with pseudocode's 1-indexed arrays is one of the most common errors on Paper 2.

Mistake 16

SQL: ORDER BY placed before WHERE

The clause order is fixed: SELECT … FROM … WHERE … GROUP BY … ORDER BY …. Putting them in a different order is a syntax error.

-- Wrong order
SELECT Name FROM Student ORDER BY Name WHERE Year = 11;
-- Right
SELECT Name FROM Student WHERE Year = 11 ORDER BY Name;
Mistake 17

Forgetting the colon in Python control statements

Every Python if, for, while, def, and else must end with a colon. Indentation alone is not enough.

# Wrong
if x > 10
    print(x)

# Right
if x > 10:
    print(x)
Mistake 18

Mixing single and double quotes inconsistently

Python accepts both 'x' and "x" for strings, but mixing them in the same string literal (e.g. "hello') is a syntax error. In pseudocode, use double quotes consistently — that is what the mark scheme uses.

Mistake 19

Confusing MOD / DIV with /

DIV is integer division (drop the remainder). MOD is the remainder. / is true (float) division. Python equivalents: // and %. Mixing these up turns "is it even?" into nonsense.

7 DIV 2 = 3      // integer quotient
7 MOD 2 = 1      // remainder
7 / 2     = 3.5    // real division
Mistake 20

Not ending with an OUTPUT

If a question says "calculate X" or "find Y", the marker almost always expects you to display the result. A program that calculates the right answer but never outputs it loses 1 mark every time.

Exam Strategy

  • First 3 minutes: read the whole paper. Flag any question that looks hard with a tiny star — you'll do those last so you don't burn time staring at them.
  • Use pencil for flowcharts and trace tables. Mistakes are inevitable; ink corrections look messy and lose method-mark legibility.
  • Long answers (programming questions worth 6+ marks): begin with a one-line comment stating the plan, e.g. // initialise total, loop 10 times reading input, output total. The examiner gives method marks for clear intent even if the code has slips.
  • Trace tables: draw a thin horizontal line every time the loop restarts. It prevents you from skipping rows or duplicating values when there are several iterations to record.
  • Always end programming answers with OUTPUT (or print()). Calculating the right value but never displaying it is a near-universal mark deduction.
  • Running out of time? Write a pseudocode skeleton: INPUT → IF or LOOP → the calculation → OUTPUT. Even a 4-line skeleton typically scores 2–3 method marks — far better than zero.
  • Never leave a question blank. If you have any idea, write it. The mark scheme rewards "shows attempt to use a loop", "identifies need for validation", etc., even on partly-wrong answers.
  • For SQL: check clause order before you write the next clause. WHERE before GROUP BY before ORDER BY. Single statement, ends with a semicolon.
  • For Python answers: remember the colon, remember to indent, remember to cast inputs. Three free marks lost there in a typical paper.
  • State the validation type by name (range / length / type / presence / format / check digit). Markers look for the word.
  • After every section, glance back at the question. Did it ask for position or value? Count or list? Many slips come from answering a slightly different question.
