Program Development Life Cycle

The structured, iterative process used by programmers to turn a real-world problem into a working, tested program — examined heavily on Paper 2.

The Four Stages at a Glance

Stage 1
Analysis

Understand the problem. Gather requirements. Identify inputs, processes, outputs and storage.

Stage 2
Design

Plan the solution using structure diagrams, flowcharts and pseudocode.

Stage 3
Coding

Translate the design into a high-level programming language (Python, Java, VB).

Stage 4
Testing

Run the program with carefully chosen test data to find and fix errors.

Stage 1 — Analysis (in depth)

Analysis

Analysis is about understanding the problem fully before writing any code. It involves four key activities:

  • Requirements gathering — interview the client/user, study the existing system, read documents, identify what the program must do (functional requirements) and how well it must do it (non-functional, e.g. speed, security).
  • Abstraction — keeping only the relevant details and ignoring/hiding the rest. E.g. for a payroll program you keep pay rate and hours worked; you ignore the employee's eye colour.
  • Decomposition — breaking the big problem into smaller, manageable sub-problems that can each be solved separately.
  • Identifying Inputs / Processes / Outputs / Storage — every program is essentially I → P → O, often with files/databases for persistent storage.

Stage 2 — Design (in depth)

Design

Design takes the analysis output and produces a plan that a programmer can follow. Three standard tools are examined:

Structure diagram

A top-down, hierarchical tree showing how the whole system is broken into modules and sub-modules.

Use when: planning overall architecture at the start of design.

Flowchart

A visual diagram of the algorithm using standard symbols (start/stop, process, decision, input/output).

Use when: showing flow of control, especially loops and decisions.

Pseudocode

Text-based, language-independent description of the algorithm using keywords like IF, WHILE, FOR.

Use when: writing detailed step-by-step logic just before coding.

Sample pseudocode (login validation)

INPUT username
INPUT password
IF username = "admin" AND password = "1234" THEN
   OUTPUT "Access granted"
ELSE
   OUTPUT "Access denied"
ENDIF

Stage 3 — Coding (in depth)

Coding

Coding is the translation of the design into a high-level programming language such as Python, Java or Visual Basic. A high-level language is chosen because it is easier to read, write and maintain than machine code.

  • Modular coding — splitting the program into procedures (which carry out a task) and functions (which carry out a task and return a value). Each module corresponds to one box on the structure diagram.
  • Benefits of modules: easier to test, easier to debug, can be reused, can be written by different team members in parallel, easier to maintain.
  • Code conventions: meaningful identifiers, comments, indentation, constants for fixed values.

Example — modular pseudocode

PROCEDURE Greet(name : STRING)
   OUTPUT "Hello ", name
ENDPROCEDURE

FUNCTION Square(x : INTEGER) RETURNS INTEGER
   RETURN x * x
ENDFUNCTION

// Main program
Greet("Alex")
OUTPUT Square(7)

Stage 4 — Testing (in depth)

Testing

Testing checks that the program does what it should and copes with bad input. Four kinds of test data are defined by the syllabus.

Normal

Sensible values well inside the accepted range. Should be accepted.

Range 1–100 → 42

Boundary

Values on the edge of the accepted range. Largest accepted & smallest accepted.

Range 1–100 → 1 or 100

Extreme

The most extreme valid values at each end of the accepted range (often same as boundary).

Range 1–100 → 1 and 100

Erroneous

Values the program should reject — wrong type or out of range.

Range 1–100 → -5, 101, "abc"

Boundary vs Extreme — key distinction: Boundary data tests just inside and just outside the limit (e.g. 0, 1, 100, 101). Extreme data tests the largest and smallest values that are still accepted (1 and 100). In the IGCSE answers, the boundary just inside and the extreme can be the same number.

Iterative testing: testing is not a one-off step at the end. When a test fails, the programmer returns to coding (and sometimes design or analysis) to fix the cause, then re-tests. This loop continues until all tests pass.

The Cycle is Iterative

The four stages form a cycle: bugs send you back to coding, design flaws send you back to design, and missed requirements send you all the way back to analysis.

flowchart LR A[Analysis] --> B[Design] B --> C[Coding] C --> D[Testing] D -- bugs found --> C D -- design flaw --> B D -- requirements wrong --> A

Types of Programming Errors

Three classes of error appear on Paper 2 — make sure you can define each and give an example.

Syntax error

A break of the rules (grammar) of the programming language. Detected by the compiler or interpreter before/while the program runs — the program will not execute.

Example:

IF x > 10      // missing THEN
   OUTPUT "big"
ENDIF

Or in Python: print("hi" — missing closing bracket.

Logic error

The program runs without crashing but produces the wrong output. The instructions are valid but they do the wrong thing.

Example: an average function that uses total / 2 instead of total / n, or IF age > 18 instead of IF age >= 18.

Runtime error

The program starts running but crashes (aborts) during execution because it tries to do something impossible.

Example: division by zero (x / 0), accessing array element a[20] when array only has 10 elements, or trying to open a file that does not exist.

Exam-style Questions

Q1. State the four stages of the program development life cycle and describe what happens at each. [8]

Show model answer
  • Analysis — the problem is investigated; requirements are gathered from the user; abstraction and decomposition are applied; inputs, processes, outputs and storage are identified. [2]
  • Design — a solution is planned using tools such as structure diagrams, flowcharts and pseudocode; data structures and module interfaces are defined. [2]
  • Coding — the design is translated into a high-level programming language; the program is written as procedures/functions (modular coding). [2]
  • Testing — the program is run with normal, boundary, extreme and erroneous test data to find errors; failed tests trigger a return to coding/design (iterative). [2]

Computational Thinking

The three problem-solving skills every IGCSE candidate must be able to define, apply and exemplify: abstraction, decomposition and algorithmic thinking.

The Three Skills

Abstraction

Removing or hiding unnecessary detail so that only the information needed to solve the problem remains.

Example: the London Tube map shows stations and lines but ignores exact distances, curves and surface geography — those details are irrelevant for finding a route.

Decomposition

Breaking a large complex problem into smaller, more manageable sub-problems that can be solved one at a time.

Example: a quiz app decomposes into: user login, question bank, scoring engine, results screen and user interface.

Algorithmic thinking

Working out a clear, ordered sequence of steps (an algorithm) that solves the problem and can be repeated reliably.

Example: writing the precise steps to make a cup of tea — kettle on, wait for boil, pour into cup with teabag, wait, remove bag, add milk.

Worked Example — Vending Machine Controller

Suppose we are asked to build the software that controls a vending machine. We apply decomposition to break the problem down, then abstraction to keep only the relevant details, then algorithmic thinking for each module.

Top-level decomposition (structure diagram)

flowchart TD A[Vending Machine] --> B[Accept Money] A --> C[Select Product] A --> D[Dispense Product] A --> E[Give Change] B --> B1[Read Coin] B --> B2[Validate Coin] B --> B3[Add to Balance] C --> C1[Display Menu] C --> C2[Read Selection] C --> C3[Check Stock] D --> D1[Activate Motor] D --> D2[Update Stock] E --> E1[Calculate Change] E --> E2[Release Coins]

Abstraction in this problem

We ignore: the colour of the machine, the brand of the cans, the music it plays, the weight of the coins. We keep: coin value, product ID, product price, stock level, balance held, change owed.

Algorithm for "Accept Money" (pseudocode)

balance ← 0
WHILE balance < price DO
   INPUT coin
   IF ValidCoin(coin) = TRUE THEN
      balance ← balance + coin
   ELSE
      OUTPUT "Coin rejected"
   ENDIF
ENDWHILE
OUTPUT "Balance reached"

Each leaf box in the structure diagram becomes a small procedure or function during the coding stage — a direct link between Computational Thinking and the Program Development Life Cycle.

Structure Diagrams Explained

A structure diagram is the most common way to present decomposition on paper. Key features:

  • Top-down — the whole system sits at the top; the diagram becomes more detailed as you go down.
  • Hierarchical — boxes are organised in levels (parent → child), with each child being a sub-task of its parent.
  • Modules, not code — each box describes what is done, not how. Algorithms come later (in pseudocode/flowcharts).
  • Planning tool — used during the Design stage of the life cycle to give the team an overview before writing detailed algorithms.
  • Reusable modules — a module that appears in more than one place (e.g. "Validate Coin") can be written once and reused.

Difference from a flowchart: a structure diagram shows what modules exist and how they relate; a flowchart shows the order in which steps execute, including loops and decisions.

Q1. Define abstraction and decomposition. Give an example of each from real-world software design. [4]

Show model answer
  • Abstraction — the process of removing or hiding unnecessary detail so that only the information relevant to solving the problem remains. [1] Example: a map app stores only road junctions, road names and one-way directions; it ignores building colours and road surface materials. [1]
  • Decomposition — breaking a complex problem into smaller sub-problems that can be solved individually. [1] Example: an e-commerce site is decomposed into login, product catalogue, shopping basket, payment and order tracking modules. [1]
